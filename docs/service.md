# Service Layer Documentation

The service layer in this application is responsible for encapsulating business logic, acting as an intermediary between controllers (or handlers) and the data access layer (repositories). Services coordinate various operations, perform validations, and manage complex workflows.

## Service Location

Service files are typically organized within the `app/services/` directory. Each service is usually named following the pattern `[service_name]_service.go`, for example, `app/services/user_service.go`.

## Service Structure Example

A typical service consists of an interface (defining the contract for the business logic) and its implementation. This promotes loose coupling and testability.

Here's a simplified example reflecting the project's actual usage:

```go
package services

import (
	"context"
	"errors"

	"github.com/rachmanzz/fiber-starter/app/repository" // Assuming repository contains Querier
	"github.com/rachmanzz/fiber-starter/app/repository/contract"
)

// ExampleServiceInterface defines the contract for our example business logic.
type ExampleServiceInterface interface {
	ProcessData(ctx context.Context, id string) (string, error)
	CreateData(ctx context.Context, data string) error
}

// exampleService is the implementation of ExampleServiceInterface.
type exampleService struct {
	repo repository.Querier // Dependency on the generated SQLC Querier
}

// NewExampleService is the constructor for ExampleService.
func NewExampleService() ExampleServiceInterface {
	return &exampleService{
		repo: contract.GetQueries(), // Instantiate Querier using the contract
	}
}

// ProcessData implements the business logic to process data.
func (s *exampleService) ProcessData(ctx context.Context, id string) (string, error) {
	// Example: Fetch data from repository using the Querier interface
	data, err := s.repo.GetData(ctx, id) // Assuming GetData is part of repository.Querier
	if err != nil {
		return "", errors.New("failed to retrieve data")
	}
	// ... perform other business logic ...
	return "Processed: " + data, nil
}

// CreateData implements the business logic to create new data.
func (s *exampleService) CreateData(ctx context.Context, data string) error {
	// Example: Validate data, then save to repository using the Querier interface
	if data == "" {
		return errors.New("data cannot be empty")
	}
	return s.repo.SaveData(ctx, data) // Assuming SaveData is part of repository.Querier
}
```

In this example:
- `repository.Querier` is the actual interface for interacting with the database, typically generated by SQLC.
- `ExampleServiceInterface` defines what operations the service offers.
- `exampleService` is the concrete implementation, holding a `repository.Querier` dependency.
- `NewExampleService` is the constructor for the service, where `contract.GetQueries()` is used to obtain an instance of `repository.Querier`.

**Note:** The module path `github.com/rachmanzz/fiber-starter` used in the imports above should be replaced with your actual module name after running `./spark init`.

## Flexibility in Structure

This documentation outlines a recommended and common practice for structuring services within this boilerplate. However, this structure is not absolute. Developers are encouraged to adapt and evolve the service architecture based on their project's specific needs, team conventions, and industry best practices, as long as it maintains clarity, testability, and maintainability.
